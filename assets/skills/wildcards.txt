wildcards

Purpose:
Define and use reusable prompt fragments and prompt-generation tokens. Assets are external project resources, not memory and not identity.

When to use:
- When the user asks for wildcard lists, prompt fragments, prompt batches, style libraries, clothing lists, pose lists, camera lists, lighting lists, templates, or reusable generation fragments.
- When the user asks to assemble or generate prompts from reusable fragment tokens.
- Use project assets instead of pasting huge reusable lists into chat.

Folder layout:
- assets/wildcards/ contains reusable prompt fragment lists.
- assets/prompts/ contains ready prompt batches.
- assets/templates/ contains prompt templates.
- assets/outputs/ contains generated or assembled outputs.

Wildcard files:
- Plain .txt only.
- One line equals one reusable prompt fragment.
- No markdown, numbering, JSON, comments, quotes, or decorative headings inside wildcard files.
- Read only the needed file or a small sample. Do not inline large wildcard libraries into chat context.

Wildcard syntax:
- __category/file__ means read one random line from assets/wildcards/category/file.txt.
- The token path must match a real wildcard file path without the .txt extension.
- Example: if list_wildcards returns assets/wildcards/clothing/tops.txt, the valid token is __clothing/tops__.
- Examples are illustrative only. Never treat example names as existing project files.

Path authority:
- Wildcard filenames are project data, not naming conventions.
- Never infer, invent, normalize, translate, or improve wildcard paths from meaning.
- Do not guess names like women_tops, women_bottoms, shoe_types, poses_list, etc. unless those exact files were created in the current workflow or returned by list_wildcards.
- Existing wildcard paths are valid only if they came from:
  1. list_wildcards result,
  2. sample_wildcard / preview_file result,
  3. a successful asset action in the current workflow that created that exact path.
- If a needed path was not verified, call list_wildcards before using it.
- If the file is not listed and was not just created successfully, treat it as missing. Do not substitute a guessed path.

Action workflow:
1. Use LIST_SKILLS to retrieve relevant project skills before a wildcard workflow.
2. Before any action that references existing wildcard files, call list_wildcards.
3. After list_wildcards, use only the exact returned wildcard paths, without .txt, inside __category/file__ tokens.
4. Use ASSET_ACTION with JSON payload for list_wildcards, sample_wildcard, expand_template, generate_prompt_batch, check_duplicates, or preview_file.
5. Emit ASSET_ACTION as a JSON block:
<ASSET_ACTION>
{"action":"list_wildcards"}
</ASSET_ACTION>
6. Payload fields may be top-level or nested under args.
7. When creating or updating reusable fragment resources, keep the wildcard file format and path authority rules active.
8. Use generate_prompt_batch when the user asks to create N prompts from a template or wildcard tokens and save them to assets/prompts or assets/outputs.
   Schema:
   {"action":"generate_prompt_batch","template":"...","count":20,"path":"assets/prompts/test_prompts.txt"}
9. Before generate_prompt_batch:
   - identify every __category/file__ token needed by the template;
   - verify every existing token through list_wildcards;
   - use newly created wildcard paths only if their create action succeeded;
   - never use guessed wildcard paths.
10. Prompt batch outputs must contain fully expanded prompts. Never write unresolved __category/file__ tokens as final prompt lines.
11. If a required wildcard file from the template is missing, stop and report the missing wildcard path, unless the user explicitly asked to create that resource first.
12. After ASSET_ACTION completes, report paths, counts, and 3-5 examples when useful. Do not paste huge generated lists into chat.

Correct example:
- If list_wildcards shows:
  assets/wildcards/clothing/tops.txt
  assets/wildcards/clothing/bottoms.txt
- Then valid tokens are:
  __clothing/tops__
  __clothing/bottoms__

Wrong:
- __clothing/women_tops__
- __clothing/women_bottoms__

Reason:
Those paths look meaningful, but they are not valid unless the files actually exist.
